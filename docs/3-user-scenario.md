## 문서 개요

본 문서는 UC(유스케이스) 단위가 아닌 **실제 사용자의 목적 달성 흐름** 단위로 시나리오를 구성한다.
각 시나리오는 여러 UC를 조합한 연속 행동 흐름으로 서술하며, 사용자가 무엇을 이루고자 하는지를 중심에 둔다.

## v1 릴리즈

---

### US-01: 신규 사용자의 첫 사용

- **시나리오 ID**: US-01
- **제목**: 신규 사용자가 가입 후 로그인하여 첫 할일을 등록하는 흐름
- **관련 UC**: UC-001, UC-002, UC-008
- **페르소나**: 처음 서비스에 접속한 대학생 또는 직장인
- **목표**: 회원가입을 완료하고, 로그인 후 첫 번째 할일을 등록한다.
- **사전 조건**: 사용자가 앱에 처음 접속한 미인증 상태이다.

**흐름**:

1. 사용자가 회원가입 화면에서 이름, 이메일, 비밀번호(8자 이상)를 입력하고 가입을 요청한다.
2. 시스템이 이메일 중복 여부를 확인하고, 계정을 생성한다. 기본 카테고리가 자동 생성된다.
3. 시스템이 로그인 화면으로 이동시킨다.
4. 사용자가 이메일과 비밀번호를 입력하고 로그인을 요청한다.
5. 시스템이 자격증명을 검증하고 JWT 토큰을 발급한다. 저장된 language·themeMode 설정과 함께 메인 화면(할일 목록)으로 이동한다.
6. 사용자가 할일 등록 버튼을 눌러 등록 화면으로 이동한다.
7. 사용자가 제목, 시작일자, 종료일자를 입력하고 등록을 요청한다. (카테고리 미지정)
8. 시스템이 기본 카테고리를 자동 적용하여 할일을 저장하고 목록에 즉시 반영한다.

- **완료 조건**: 첫 번째 할일이 목록에 표시되고, 상태가 날짜 기준으로 올바르게 표시된다.

**예외/대안 흐름**:

- 2단계: 이미 가입된 이메일 입력 시 → HTTP 409, 오류 메시지 표시 후 재입력 유도
- 2단계: 비밀번호 8자 미만 입력 시 → HTTP 400, 오류 메시지 표시
- 5단계: 이메일 또는 비밀번호 불일치 시 → HTTP 401, 오류 메시지 표시
- 7단계: 종료일자 < 시작일자 입력 시 → HTTP 422, 날짜 오류 메시지 표시

### US-02: 카테고리 체계를 구성하고 할일을 분류 등록하는 흐름

- **시나리오 ID**: US-02
- **제목**: 카테고리를 만들고 할일을 해당 카테고리에 분류하여 등록하는 흐름
- **관련 UC**: UC-005, UC-008, UC-013
- **페르소나**: 업무/개인 할일을 카테고리로 분리해서 관리하고 싶은 직장인
- **목표**: 카테고리를 신규 생성하고, 해당 카테고리로 할일을 분류 등록한 뒤 카테고리 필터로 확인한다.
- **사전 조건**: 사용자가 로그인된 상태이며, 기본 카테고리만 존재한다.

**흐름**:

1. 사용자가 카테고리 관리 화면에서 카테고리명(최대 100자)을 입력하고 등록을 요청한다.
2. 시스템이 해당 사용자 소속으로 카테고리를 생성하고 목록에 즉시 반영한다.
3. 사용자가 할일 등록 화면으로 이동하여 제목, 시작일자, 종료일자를 입력하고 방금 생성한 카테고리를 선택한다.
4. 시스템이 선택한 카테고리로 할일을 저장하고 목록에 반영한다.
5. 사용자가 할일 목록 화면에서 해당 카테고리 필터를 선택한다.
6. 시스템이 해당 카테고리에 속한 할일만 필터링하여 표시한다.

- **완료 조건**: 카테고리 필터 적용 시 등록한 할일이 해당 카테고리 아래에 정확히 표시된다.

**예외/대안 흐름**:

- 1단계: 카테고리명 100자 초과 입력 시 → HTTP 400, 오류 메시지 표시
- 5단계: 해당 카테고리에 할일이 없을 경우 → 빈 목록 표시

### US-03: 오늘 해야 할 일을 파악하고 완료 처리하는 흐름

- **시나리오 ID**: US-03
- **제목**: 진행 중인 할일을 조회하고 완료 처리하는 흐름
- **관련 UC**: UC-015, UC-011, UC-016
- **페르소나**: 하루 시작에 오늘 처리할 일을 확인하고 하나씩 완료하는 직장인 또는 학생
- **목표**: 진행 중 상태의 할일 목록을 확인하고, 완료한 항목을 완료 처리한다.
- **사전 조건**: 사용자가 로그인된 상태이며, 시작일자 <= 오늘(KST) <= 종료일자인 미완료 할일이 존재한다.

**흐름**:

1. 사용자가 할일 목록 화면에서 '진행 중' 필터를 선택한다.
2. 시스템이 isCompleted = false이고 startDate <= 오늘(KST) <= endDate인 할일을 반환하여 표시한다.
3. 사용자가 처리 완료한 할일의 완료 버튼을 클릭한다.
4. 시스템이 해당 할일의 isCompleted를 true로 변경하고, 상태를 '완료'로 표시한다.
5. 사용자가 '완료' 필터를 선택하여 완료 처리된 할일이 반영되었는지 확인한다.
6. 시스템이 isCompleted = true인 할일 목록을 반환하여 표시한다.

- **완료 조건**: 완료 처리한 할일이 '진행 중' 목록에서 사라지고 '완료' 목록에 표시된다.

**예외/대안 흐름**:

- 2단계: 진행 중인 할일이 없을 경우 → 빈 목록 표시
- 4단계: 완료 처리 후 실수로 다시 클릭(완료 취소) 시 → isCompleted = false로 복원, 날짜 기반으로 상태 재계산(진행 중 / 시작 전 / 기한 초과 중 해당 상태로 복원)

### US-04: 기한이 지난 할일을 찾아 수정하거나 삭제하는 흐름

- **시나리오 ID**: US-04
- **제목**: 기한 초과 할일을 조회하고 날짜를 수정하거나 불필요한 항목을 삭제하는 흐름
- **관련 UC**: UC-017, UC-009, UC-010
- **페르소나**: 밀린 일정을 정리하고 싶은 직장인 또는 학생
- **목표**: 기한이 지난 할일을 한눈에 파악하고, 연장이 필요한 항목은 날짜를 수정하고 불필요한 항목은 삭제한다.
- **사전 조건**: 사용자가 로그인된 상태이며, isCompleted = false이고 오늘(KST) > endDate인 할일이 존재한다.

**흐름**:

1. 사용자가 할일 목록 화면에서 '기한 초과' 필터를 선택한다.
2. 시스템이 isCompleted = false이고 오늘(KST) > endDate인 할일 목록을 반환하여 표시한다.
3. 사용자가 일정 연장이 필요한 할일을 선택하고 수정 화면으로 이동한다.
4. 사용자가 종료일자를 오늘 이후로 변경하고 저장을 요청한다.
5. 시스템이 유효성 검증 후 변경 내용을 저장한다. 날짜 조건에 따라 상태가 '진행 중' 또는 '시작 전'으로 갱신된다.
6. 사용자가 더 이상 필요 없는 할일을 선택하고 삭제를 요청한다.
7. 시스템이 해당 할일을 삭제하고 목록에서 즉시 제거한다.

- **완료 조건**: 수정한 할일이 '기한 초과' 목록에서 사라지고 갱신된 날짜 기준의 상태로 변경된다. 삭제한 할일은 목록에서 완전히 제거된다.

**예외/대안 흐름**:

- 2단계: 기한 초과 할일이 없을 경우 → 빈 목록 표시
- 4단계: 새 종료일자 < 시작일자 입력 시 → HTTP 422, 날짜 오류 메시지 표시 후 재입력 유도

### US-05: 전체 할일 현황을 필터로 점검하는 흐름

- **시나리오 ID**: US-05
- **제목**: 전체 및 상태별 필터로 할일 현황을 파악하는 흐름
- **관련 UC**: UC-012, UC-013, UC-014, UC-015, UC-016, UC-017
- **페르소나**: 주기적으로 전체 일정 현황을 점검하는 직장인 또는 학생
- **목표**: 전체 할일 및 상태별 필터를 통해 현재 진행 현황을 한눈에 파악한다.
- **사전 조건**: 사용자가 로그인된 상태이며, 다양한 상태의 할일이 존재한다.

**흐름**:

1. 사용자가 할일 목록 화면에서 '전체' 필터를 선택한다.
2. 시스템이 해당 사용자의 모든 할일을 반환하여 표시한다.
3. 사용자가 '시작 전' 필터를 선택하여 아직 시작하지 않은 할일을 확인한다.
4. 시스템이 isCompleted = false이고 오늘(KST) < startDate인 할일을 반환한다.
5. 사용자가 '진행 중' 필터를 선택하여 현재 진행 중인 할일을 확인한다.
6. 시스템이 isCompleted = false이고 startDate <= 오늘(KST) <= endDate인 할일을 반환한다.
7. 사용자가 '기한 초과' 필터를 선택하여 처리가 필요한 밀린 항목을 확인한다.
8. 시스템이 isCompleted = false이고 오늘(KST) > endDate인 할일을 반환한다.
9. 사용자가 특정 카테고리 필터를 선택하여 해당 카테고리의 할일만 확인한다.
10. 시스템이 선택한 카테고리에 속한 할일만 필터링하여 반환한다.

- **완료 조건**: 각 필터 적용 시 해당 조건에 맞는 할일 목록이 정확하게 표시된다.

**예외/대안 흐름**:

- 각 필터 적용 시 해당 조건의 할일이 없을 경우 → 빈 목록 표시

### US-06: 계정 정보 및 카테고리를 정리하는 흐름

- **시나리오 ID**: US-06
- **제목**: 이름·비밀번호를 변경하고 불필요한 카테고리를 정리하는 흐름
- **관련 UC**: UC-004, UC-006, UC-007, UC-003
- **페르소나**: 개인 설정을 변경하거나 카테고리 구조를 재편하고 싶은 사용자
- **목표**: 계정 정보(이름 또는 비밀번호)를 수정하고, 더 이상 쓰지 않는 카테고리를 수정하거나 삭제한 뒤 로그아웃한다.
- **사전 조건**: 사용자가 로그인된 상태이며, 일반 카테고리(기본 카테고리 외)가 1개 이상 존재한다.

**흐름**:

1. 사용자가 내 정보 수정 화면으로 이동하여 이름 또는 비밀번호를 변경하고 저장을 요청한다.
2. 시스템이 변경 내용을 저장하고 수정 완료 메시지를 표시한다.
3. 사용자가 카테고리 관리 화면으로 이동하여 이름을 변경할 카테고리를 선택하고 새 이름을 입력한 뒤 수정을 요청한다.
4. 시스템이 카테고리명을 업데이트하고 변경된 이름을 목록에 반영한다.
5. 사용자가 삭제할 카테고리를 선택하고 삭제를 요청한다.
6. 시스템이 해당 카테고리를 삭제한다. 하위 할일이 존재하면 기본 카테고리로 자동 재지정한다. 카테고리 목록에서 제거된다.
7. 사용자가 로그아웃을 요청한다.
8. 시스템이 발급된 토큰을 만료 처리하고 로그인 화면으로 이동한다.

- **완료 조건**: 계정 정보가 변경되고, 카테고리 목록이 정리된 상태로 로그아웃이 완료된다.

**예외/대안 흐름**:

- 1단계: 비밀번호 8자 미만 입력 시 → HTTP 400, 오류 메시지 표시
- 3단계: 기본 카테고리 수정 시도 시 → HTTP 422, "기본 카테고리는 수정할 수 없습니다." 메시지 표시
- 5단계: 기본 카테고리 삭제 시도 시 → HTTP 422, "기본 카테고리는 삭제할 수 없습니다." 메시지 표시

### US-07: 언어와 테마를 설정하고 다음 로그인 시 유지되는 흐름

- **시나리오 ID**: US-07
- **제목**: 언어 및 테마를 변경하고 재로그인 후에도 설정이 유지되는 흐름
- **관련 UC**: UC-018, UC-019, UC-003, UC-002
- **페르소나**: 다국어 환경에서 일하는 직장인 또는 다크 모드를 선호하는 사용자
- **목표**: UI 언어와 테마(다크/라이트)를 원하는 값으로 설정하고, 로그아웃 후 재로그인 시 설정이 자동으로 복원되는 것을 확인한다.
- **사전 조건**: 사용자가 로그인된 상태이며, 현재 언어 설정은 한국어(ko), 테마는 라이트(light) 상태이다.

**흐름**:

1. 사용자가 설정 화면으로 이동하여 언어를 영어(en) 또는 일본어(ja)로 선택한다.
2. 시스템이 선택값을 User.language(DB)에 저장하고 UI 언어를 즉시 변경한다. 페이지 새로고침 없이 반영된다.
3. 사용자가 테마를 다크 모드(dark)로 선택한다.
4. 시스템이 선택값을 User.themeMode(DB)에 저장하고 UI 테마를 즉시 변경한다.
5. 사용자가 로그아웃을 요청한다.
6. 시스템이 토큰을 만료 처리하고 로그인 화면으로 이동한다.
7. 사용자가 이메일과 비밀번호를 입력하고 재로그인을 요청한다.
8. 시스템이 인증 후 저장된 language·themeMode 설정을 응답에 포함하여 반환한다. 메인 화면이 저장된 언어 및 테마(다크 모드)로 표시된다.

- **완료 조건**: 재로그인 후 이전에 설정한 언어와 테마가 자동으로 적용된 상태로 메인 화면이 표시된다.

**예외/대안 흐름**:

- 1단계: 지원하지 않는 언어 코드 전달 시 → HTTP 400, 오류 메시지 표시
- 3단계: 유효하지 않은 themeMode 값 전달 시 → HTTP 400, 오류 메시지 표시
- 7단계: 이메일 또는 비밀번호 불일치 시 → HTTP 401, 오류 메시지 표시
